# **API CALL & ASYNC HANDLING** {#api-call-&-async-handling}

## ***Pentingnya Desain Responsif pada Mobile*** {#pentingnya-desain-responsif-pada-mobile}

Dalam pengembangan aplikasi modern, sebagian besar data tidak disimpan secara lokal di perangkat, tetapi diperoleh melalui komunikasi jaringan. Komunikasi ini menggunakan API (Application Programming Interface), yaitu jembatan yang memungkinkan aplikasi client berinteraksi dengan sistem server untuk mengambil, mengirim, atau memperbarui data.

***1\. Siklus Request–Response API***

Setiap komunikasi antara aplikasi Flutter dengan server mengikuti pola request–response.  
Prosesnya dapat digambarkan dalam tiga tahap utama:

1) Client mengirimkan request — aplikasi Flutter mengirimkan permintaan ke server menggunakan metode HTTP seperti GET, POST, PUT, atau DELETE.  
2) Server memproses permintaan — server membaca request, menyiapkan hasilnya (misalnya data dalam format JSON).  
3) Client menerima response — hasil tersebut diterima aplikasi, di-decode, dan ditampilkan ke antarmuka pengguna (UI).  
   

Sebagai contoh, API publik berikut mengembalikan daftar negara:

Contoh hasil response:

Setiap objek di dalam array berisi informasi nama negara yang dapat kita tampilkan di aplikasi.

***2\. REST API dan Konsep Data JSON***

Sebagian besar komunikasi API modern menggunakan REST (Representational State Transfer) dan format JSON (JavaScript Object Notation).  
REST memanfaatkan protokol HTTP standar, sementara JSON digunakan karena ringan dan mudah dibaca manusia maupun mesin.  
Ciri utama REST API:

* Menggunakan URL sebagai endpoint data (/users, /products, /countries).  
* Menggunakan metode HTTP (GET, POST, PUT, DELETE).  
* Mengembalikan data dalam format JSON.

Flutter tidak membaca JSON secara langsung. Data harus di-decode menggunakan dart:convert agar bisa diakses dalam bentuk objek atau map.

***3\. Perbandingan REST API dan GraphQL (sekilas)***

**Perbandingan Singkat**

| Aspek | REST API | GraphQL |
| ----- | ----- | ----- |
| **Struktur** | Endpoint tetap | Query fleksibel |
| **Volume Data** | Semua data dikirim sekaligus | Hanya data yang diminta |
| **Kompleksitas** | Lebih sederhana | Lebih kompleks, butuh server khusus |

Pada pembelajaran ini, digunakan REST API karena mudah dipahami dan umum digunakan dalam proyek Flutter.

## ***Dasar Asynchronous Programming*** {#dasar-asynchronous-programming}

Bagian ini menjelaskan bagaimana Flutter mengelola proses non-blocking agar UI tetap responsif walau ada tugas berat seperti memanggil API atau membaca file besar.

***1\. Konsep Dasar Future***

Future di Dart adalah representasi dari nilai yang akan tersedia di masa depan. Ketika fungsi async dijalankan, ia langsung dikembalikan sebagai Future, sementara proses di dalamnya berjalan di latar belakang (background thread).

Contoh sederhana:

![][image11]  
Ketika fungsi ini dipanggil, Flutter tidak menunggu dua detik secara pasif, tetapi melanjutkan proses lain seperti menggambar UI. Begitu delay selesai, nilai dikirim kembali ke kode pemanggil.

***2\. Event Loop: Jantung Asynchronous di Dart***

Flutter (melalui Dart runtime) menggunakan sistem event loop, yaitu antrean (queue) yang menangani tugas secara bergantian tanpa menghentikan eksekusi utama.

Ilustrasi sederhananya:

1) Flutter memulai *main thread* untuk menggambar UI.  
2) Ketika ada tugas berat (misalnya API call), tugas itu dimasukkan ke **event queue**.  
3) Sambil menunggu hasil, Flutter tetap melanjutkan tugas lain (misalnya animasi, input pengguna).  
4) Ketika hasil Future sudah siap, event loop mengambilnya dari antrean dan menjalankan callback yang sesuai.

Itulah sebabnya meski menunggu data dari server, aplikasi tidak mengalami freeze.

***3\. async-await vs Callback***

Dart menyediakan dua cara umum untuk mengelola Future:

***a. Pendekatan async–await***  
Lebih mudah dibaca, menyerupai urutan logika sinkron.

![][image12]

***b. Pendekatan callback chaining***  
Menggunakan fungsi .then() dan .catchError().

![][image13]

**Perbandingan**

| Aspek | async-await | Callback |
| ----- | ----- | ----- |
| **Keterbacaan** | Linear dan bersih | Mudah kacau jika bertingkat |
| **Debugging** | Lebih mudah dilacak | Susah diikuti |
| **Performa** | Setara | Setara |
| **Umum digunakan** | Ya (kode modern) | Tidak direkomendasikan untuk kompleksitas tinggi |
